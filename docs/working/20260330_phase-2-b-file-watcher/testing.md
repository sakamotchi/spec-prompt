# テスト手順書 - Phase 2-B: ファイル監視・自動更新

## 概要

ファイル監視・自動更新機能のテスト手順を記載します。外部エディタでファイルを編集・保存し、アプリの表示が自動更新されることを確認します。

## 前提条件

- `npx tauri dev` でアプリが起動していること
- テスト用プロジェクトとして `spec-prompt` リポジトリ自体を開いておく
- テキストエディタ（VS Code、nano 等）が使用できること

---

## 手動テスト

### ケース 1: マークダウンファイルの自動更新

**手順:**

1. アプリのファイルツリーから任意の `.md` ファイルをクリックしてプレビューを表示する
2. 外部エディタ（VS Code 等）で同じファイルを開き、テキストを追記して保存する
3. アプリのコンテンツペインを確認する

**期待結果:**

- 保存後 500ms 以内にコンテンツペインの表示が更新される
- 追記したテキストがプレビューに反映される

**確認結果:**

- [x] OK / NG

---

### ケース 2: コードファイルの自動更新

**手順:**

1. アプリのファイルツリーから `.ts` ファイルをクリックしてコードビューアを表示する
2. 外部エディタで同じファイルを編集して保存する
3. アプリのコンテンツペインを確認する

**期待結果:**

- 保存後に変更内容がコードビューアに反映される

**確認結果:**

- [x] OK / NG

---

### ケース 3: 表示していないファイルを変更しても更新されない

**手順:**

1. ファイル A を表示している状態で、外部エディタでファイル B を編集・保存する

**期待結果:**

- ファイル A の表示は変化しない（ファイル B の変更は無視される）

**確認結果:**

- [x] OK / NG

---

### ケース 4: ファイル追加時のツリー自動更新

**手順:**

1. アプリでプロジェクトを開いた状態にする
2. ターミナル（アプリ内またはシステム）で新しいファイルを作成する
   ```bash
   touch /path/to/project/test_new_file.md
   ```
3. アプリのファイルツリーを確認する

**期待結果:**

- `test_new_file.md` がファイルツリーに表示される

**確認結果:**

- [x] OK / NG

---

### ケース 5: ファイル削除時のツリー自動更新

**手順:**

1. ケース 4 で作成した `test_new_file.md` を削除する
   ```bash
   rm /path/to/project/test_new_file.md
   ```
2. アプリのファイルツリーを確認する

**期待結果:**

- `test_new_file.md` がファイルツリーから消える

**確認結果:**

- [x] OK / NG

---

### ケース 6: `node_modules` の変更がツリーを壊さない

**手順:**

1. アプリでプロジェクトを開いた状態にする
2. ターミナルで `npm install` を実行する（`node_modules` に大量の変更が発生する）
3. アプリのファイルツリーとコンテンツビューアを確認する

**期待結果:**

- アプリがハングしない・クラッシュしない
- ファイルツリーの表示が崩れない

**確認結果:**

- [x] OK / NG

---

### ケース 7: プロジェクトを開いたら自動で監視が始まる

**手順:**

1. アプリを起動し、まだプロジェクトを開いていない状態にする
2. 「プロジェクトを開く」からフォルダを選択する
3. ファイルを外部エディタで編集・保存する

**期待結果:**

- フォルダ選択後から自動的に監視が開始され、ファイル変更が反映される

**確認結果:**

- [x] OK / NG

## エッジケース

| ケース | 期待動作 | 確認結果 |
|--------|---------|---------|
| 同じファイルを高速で連続保存（1秒に5回） | 最後の変更が反映される（デバウンスで不要な更新を抑制） | |
| 大きなファイル（10,000行以上）を保存 | 500ms 以内に更新される | |
| `.git` ディレクトリへの変更（git commit 等） | ツリーが更新されない・アプリが重くならない | |
