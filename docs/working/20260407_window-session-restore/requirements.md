# 要件定義書 - Phase 3-H: ウィンドウセッション復元

## 概要

アプリ終了時に開いていた複数ウィンドウの状態（プロジェクトパス）を保存し、次回起動時に同じウィンドウ構成を自動復元する。

## 背景・目的

Phase 3-G で複数ウィンドウ表示機能を実装したが、アプリを終了して再起動すると常に1ウィンドウのみの状態に戻ってしまう。複数のプロジェクトを並行して作業するユーザーにとって、毎回ウィンドウを手動で開き直す手間が発生している。

この機能により、作業の継続性が向上し、複数ウィンドウ機能の実用性が大きく高まる。

## 要件一覧

### 機能要件

#### F-1: ウィンドウ一覧の保存

- **説明**: ウィンドウが開かれた・閉じられたタイミングで、全ウィンドウの `projectRoot`（`null` = 空ウィンドウ）リストを localStorage に保存する
- **受け入れ条件**:
  - [ ] 新規ウィンドウが開かれると localStorage のウィンドウ一覧に追加される
  - [ ] ウィンドウが閉じられると localStorage のウィンドウ一覧から削除される
  - [ ] 保存されるキーは `specprompt-window-sessions`（固定）

#### F-2: 起動時のウィンドウ復元

- **説明**: メインウィンドウ起動時に localStorage からウィンドウ一覧を読み取り、2件目以降を `WebviewWindow` で再生成する
- **受け入れ条件**:
  - [ ] 前回セッションで2つ以上のウィンドウが開かれていた場合、起動時に同数のウィンドウが復元される
  - [ ] 各ウィンドウが保存時の `projectRoot` を引き継いで起動する（空ウィンドウは空で復元）
  - [ ] 復元に失敗したウィンドウ（プロジェクトパスが存在しない等）はスキップし、アプリはクラッシュしない

#### F-3: プロジェクトパスの検証

- **説明**: 復元時にプロジェクトパスが実際に存在するか確認し、存在しない場合は空ウィンドウとして起動する
- **受け入れ条件**:
  - [ ] 存在しないパスが保存されていた場合、空ウィンドウ（`?new=1`）として復元される
  - [ ] ユーザーにエラーダイアログは表示しない（サイレントにフォールバック）

### 非機能要件

- **パフォーマンス**: 復元処理はバックグラウンドで行い、メインウィンドウの起動を遅延させない
- **ユーザビリティ**: ユーザーが意識せずとも自動的に復元される。設定でのオン/オフ切り替えは本フェーズのスコープ外とする
- **保守性**: ウィンドウセッション管理ロジックを `src/lib/windowSession.ts` に分離し、`main.tsx` を肥大化させない

## スコープ

### 対象

- メインウィンドウ起動時の追加ウィンドウ復元
- ウィンドウ開閉時のセッション保存・削除
- 存在しないプロジェクトパスへのフォールバック

### 対象外

- タブ状態・ターミナルセッションの復元（Zustand persist で別途管理）
- ウィンドウの位置・サイズの復元
- 復元機能のオン/オフ設定
- アプリの強制終了時の保存（OS シグナルによる強制終了は対象外）

## 実装対象ファイル（予定）

- `src/lib/windowSession.ts`（新規 — ウィンドウセッション管理ユーティリティ）
- `src/main.tsx`（変更 — 起動時の復元処理を追加）
- `src/components/TreePanel/TreePanel.tsx`（変更 — ウィンドウ閉じイベントのフック）

## 依存関係

- Phase 3-G（複数ウィンドウ表示）— `WebviewWindow` API・URL クエリパラメータ方式を継承

## 既知の制約

- **localStorage の共有**: Tauri v2 では全ウィンドウが同一の localStorage を共有するため、複数ウィンドウが同時に書き込む場合の競合が発生しうる。初期実装では競合制御は行わず、最後に書き込んだウィンドウの状態が残る動作とする
- **強制終了時の非対応**: `Cmd+Q` や `kill` による強制終了時は `onCloseRequested` が発火しないため、セッションが保存されない場合がある
