# テスト手順書 - macos-window-tabs-f1

F1 で担保する **受け入れ基準 2, 3, 10**（元要件 `docs/local/20260418-macOSウィンドウタブ統合/01_要件定義書.md` §7）の手動確認と、自動テストの実行方法をまとめる。

## 概要

F1 は `tauri.conf.json` と `tauriApi.ts:openNewWindow` の 2 箇所に `tabbingIdentifier` を追加するだけの極小フェーズ。主要検証は OS 機能（Merge All Windows）を使った手動 E2E。

**重要**：ウィンドウ生成は既存の「ツリーヘッダの新規ウィンドウボタン」（`src/components/TreePanel/TreePanel.tsx:242`）で実施する。`npx tauri dev` は 1 プロセスで複数ウィンドウを生成でき、同一 Tauri プロセス内のウィンドウ間で NSWindow タブ統合が機能する。

## 前提条件

- `feature/macos-window-tabs` ブランチをチェックアウト済み
- `npm install` 済み
- `cd src-tauri && cargo clean` の上 `npx tauri build` 済み（conf.json の変更反映のため初回はフルビルド推奨）
- `.app` は Finder から起動（`/Applications/` と `target/release/bundle/macos/` が両方あると事故の元。どちらか一方に絞る）
- macOS で検証（F1 の受け入れ基準は macOS 限定。Windows / Linux は F4 で別途確認）
- テスト用プロジェクトフォルダを 2 つ用意（例: `~/tmp/proj-a`, `~/tmp/proj-b`）

### macOS システム設定の扱い

システム設定「デスクトップと Dock > ウィンドウ > 書類をタブで開く」の値で、タブ統合の挙動が変わります：

| 設定値 | 挙動 |
|---|---|
| **常に (Always)** | 2 つ目のウィンドウ生成時に**自動統合**される（VS Code と同等） |
| **フルスクリーン時のみ** | 非全画面時は独立、全画面時は自動統合（Tauri Issue #6548 の環境） |
| **しない (Never)** | 自動統合しない。View > Show Tab Bar またはドラッグで**手動統合のみ** |

F1 の受け入れ基準 2 は「自動 or 手動のいずれかでタブ統合できる」を達成条件とし、システム設定に依存しないので**方法 A（自動）と方法 B（手動）のどちらか片方が成功すれば OK** とする。

Tauri v2 は `NSWindow.tabbingMode` を露出しないため、VS Code のように設定を無視して常に自動タブ化する挙動は v1.1（F5）以降の候補（objc2 経由で実装する）。

## 手動テスト

### ケース 1: 2 つ目のウィンドウ起動でタブ統合される（受け入れ基準 2）

**方法 A: 自動統合（システム設定「常に」の場合）**

1. `npx tauri build` で生成した `.app` を Finder から起動する。
2. プロジェクト A（`~/tmp/proj-a`）を開く。
3. ツリーヘッダ右側の「新規ウィンドウ」ボタン（`src/components/TreePanel/TreePanel.tsx:242`）をクリックする。
4. 2 つ目のウィンドウが開いたら、プロジェクト B（`~/tmp/proj-b`）を開く。

**方法 B: 手動統合（システム設定「しない」「フルスクリーン時のみ」の場合）**

1. 方法 A の 1〜4 を実施（独立ウィンドウ 2 つになる）。
2. どちらかをアクティブにして **View > Show Tab Bar** でタブバーを表示。
3. もう一方のウィンドウのタイトルバーをタブバーへドラッグ&ドロップして統合。
4. もしくは Window メニューに出ていれば **Merge All Windows** を実行。

**期待結果:**

- タブバーが表示される（方法 A は自動、方法 B は Show Tab Bar で明示表示）
- タブが 2 つ出現し、それぞれプロジェクト名が付いている
- View メニューに **Show Tab Bar** が表示されている（これ自体が `tabbingIdentifier` 有効化の証拠）

**確認結果:**

- [x] OK（2026-04-18 確認、方法 B で統合できることを検証済み）
- 方法 A は未検証（システム設定次第で成立）

### ケース 2: タブ名にプロジェクト名が反映される（受け入れ基準 3）

**手順:**

1. ケース 1 の状態で、タブバーを確認する。
2. 各タブの名前を目視確認する。

**期待結果（理想）:**

- タブ 1 の名前: `proj-a`
- タブ 2 の名前: `proj-b`

**現行タイトル書式の仕様確認:**

現行の `TreePanel.tsx:54` は `setTitle(`SpecPrompt — ${name}`)` の書式。macOS NSWindow は `—` 前方をタブ名に使うため、タブ名が `SpecPrompt`（全タブ同名）になる可能性がある。

- **もしタブ名が `SpecPrompt` 固定になった場合**: `TreePanel.tsx:54` を `setTitle(`${name} — SpecPrompt`)` に反転し、再度確認する（tasklist T-4b）。
- **タブ名にプロジェクト名が出ていれば**: 書式変更は不要（tasklist T-4a）。

**確認結果:**

- [ ] OK（プロジェクト名が出ている）
- [ ] NG → 書式反転で対応、再確認後 OK
- 実施した対応: （T-4a / T-4b）

## 補足

- **Windows / Linux 確認は F4 に先送り**。F1 の受け入れ基準は macOS 限定。
- `tabbingIdentifier` は macOS 以外では無視されるため、Windows / Linux での回帰は「既存の新規ウィンドウ機能が従前通り動く」ことの確認で足りる。
- 現行タイトル書式が `SpecPrompt — ${name}`（SpecPrompt 先頭）のため、macOS の NSWindow タブ名切り詰めルール（`—` 後方を切る）と合わない可能性がある。ケース 2 で判定し、必要なら書式反転（T-4b）を別コミットで入れる。
