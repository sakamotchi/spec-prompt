# 要件定義書 - file-tree-dnd

## 概要

プロジェクトツリーに対し、以下 2 種類のドラッグ＆ドロップ（DnD）操作を導入する。

1. **ツリー内 DnD**: ツリー上のファイル／フォルダを別フォルダへドロップして **移動** する。
2. **Finder からの外部 DnD**: macOS Finder からのファイル／フォルダドロップを受け取り、ツリー上の対象フォルダへ **コピー** する。

両者は UI 上で同じドロップターゲット表現に統合する。

## 背景・目的

- 現状はツリーから既存ファイルを別フォルダへ移すには、右クリック → リネームでパスを書き換えるか、ターミナルから `mv` を打つ必要があり、章立て再編の局面で煩雑。
- 外部からのファイル取り込み手段がなく、Finder と SpecPrompt の往復が摩擦になっている。仕様駆動開発で図版・テンプレート等を取り込む頻度を考えると、無視できないコスト。
- 既存の右クリックメニュー・インライン入力など基本操作は揃っており、DnD は次の自然な拡張。
- 元仕様は `docs/local/20260415-ファイルドラッグドロップ操作/01_要件定義書.md` および `02_概要設計書.md`。

## 要件一覧

### 機能要件

#### F-1: ツリー内 DnD による移動

- **説明**: ツリーノードをドラッグして別フォルダノード（またはルート領域）にドロップすると、`rename_path` 経由でドロップ先配下へ移動する。複数選択中はその全件を一括移動する。
- **受け入れ条件**:
  - [ ] 単一ノードを別フォルダへドロップして移動できる
  - [ ] 複数選択中（Ctrl/Cmd+クリック等で選択した状態）の全件を一括移動できる
  - [ ] ドラッグ元と同じ親フォルダへのドロップは no-op
  - [ ] フォルダを自身または子孫配下へドロップしようとすると禁止される（カーソル not-allowed、ハイライトなし）
  - [ ] 同名衝突時はエラートーストを出して移動を中断する
  - [ ] 移動完了後、ドラッグ元・ドロップ先の両ディレクトリが再描画される
  - [ ] 移動対象がコンテンツタブで開かれている場合、タブのパスが新パスに追随する
  - [ ] 選択中ファイル（`selectedFile`）が移動された場合、新パスに追随する

#### F-2: Finder からの外部 DnD によるコピー

- **説明**: macOS Finder（および他アプリの OS DnD ソース）からツリー上のフォルダノードへドロップすると、新規 `copy_path` コマンドで再帰コピーする。
- **受け入れ条件**:
  - [ ] Finder のファイル単体をドロップしてコピーできる
  - [ ] Finder の複数ファイルをドロップして全件コピーできる
  - [ ] フォルダをドロップすると階層ごと再帰コピーされる
  - [ ] コピー後も Finder 側の元ファイルが残る（移動ではなくコピーになっている）
  - [ ] 同名衝突時は該当項目をスキップし、最後にスキップ件数をトースト通知する
  - [ ] プロジェクト未オープン時はドロップを受け付けない（カーソル not-allowed）
  - [ ] ツリー外（コンテンツビュー、ターミナル）にドロップした場合は何もしない

#### F-3: ドロップ先の視覚フィードバック

- **説明**: ドラッグ中、ドロップ可能なフォルダノードとルート領域をハイライト表示する。内部 DnD と外部 DnD で表現を共通化する。
- **受け入れ条件**:
  - [ ] ドロップ可能なフォルダノードがドラッグオーバー時にハイライトされる
  - [ ] プロジェクトルート直下にドロップ可能な余白領域もハイライトされる
  - [ ] 自己／子孫など禁止位置にはハイライトを出さない
  - [ ] 内部 DnD でドラッグ中のノード自身は薄く表示される（opacity ダウン）

#### F-4: 確認・進捗・結果通知

- **説明**: 件数や規模に応じて確認ダイアログを表示し、完了時にトーストで結果を通知する。
- **受け入れ条件**:
  - [ ] 内部移動が複数件のときに確認ダイアログを表示する
  - [ ] 外部コピーで合計サイズが大きい（目安 100MB 超）または 50 ファイル超のときに確認ダイアログを表示する
  - [ ] 完了時に件数を含むトーストを表示する
  - [ ] 部分失敗時は成功件数とエラー件数のサマリを表示する

### 非機能要件

- **パフォーマンス**:
  - 内部移動（rename）は同一ファイルシステム前提で 1 件 100ms 以内を目標。
  - 外部コピーは Rust 側で `tokio::task::spawn_blocking` を使い UI スレッドを止めない。
- **ユーザビリティ**:
  - 既存のクリック・ダブルクリック・コンテキストメニュー操作を破壊しない。5px 程度のドラッグ閾値を設けて誤発火を防ぐ。
- **保守性**:
  - 内部 DnD と外部 DnD の判定・実行を `useTreeDnd` フックに集約し、`TreeNode`／`TreePanel` のロジックを薄く保つ。
- **外観・デザイン**:
  - ハイライトは既存カラーパレット（`--color-bg-elevated` ベース + `--color-accent` 混色）で表現し、ダーク／ライトテーマの両方で視認性を担保する。

## スコープ

### 対象

- macOS（プロジェクト全体の v1 スコープと整合）
- ツリー内移動／Finder からのコピー／視覚フィードバック／結果通知

### 対象外

- ツリーから Finder 等への逆方向 DnD（外部書き出し）
- 修飾キーによる操作切替（例: Option で内部 DnD をコピーへ切替）
- 同名衝突時の自動リネーム（`foo (1).md` 等）
- 進行中操作のキャンセル
- ツリー外領域への外部ファイルドロップ（コンテンツビューに開く等）
- DnD でのファイル並び順制御（並び順は既存どおり）
- Windows / Linux 対応の検証

## 依存関係

- Tauri v2 Webview API: `getCurrentWebview().onDragDropEvent`
- 既存 `rename_path`（`src-tauri/src/commands/filesystem.rs:63`）を内部移動に流用
- 既存 capability `fs:read-all` / `fs:write-all`（追加権限不要）
- 既存ストア `appStore`（`updateDirChildren`、選択状態）、`contentStore`（`renameTabPath`）

## 既知の制約

- `dragDropEnabled: true` に切り替えると、OS 由来の HTML5 file drop イベントは webview に届かず Tauri 側で捕捉される。今後コンテンツビュー等で外部ファイル DnD を受けたい場合は `onDragDropEvent` 経由で実装する必要がある。
- 外部 DnD のドロップ先判定は `elementFromPoint` による DOM ヒットテスト。ツリー仮想化を導入した場合は再検証が必要。
- フォルダ再帰コピー中に元ディレクトリが変化した場合の整合は完了後の `updateDirChildren` で吸収する（ベストエフォート）。
- `fs::rename` がクロスデバイス（EXDEV）になる場合は v1 ではエラー表示にとどめる。
