# テスト手順書 - prompt-palette-f1

F1（MVP）で担保する **受け入れ基準 1, 2, 3, 4, 5**（元要件 §7）の手動確認と、自動テストの実行方法をまとめる。当初 14 も F1 対象だったが、2026-04-18 の E2E でケース 8 が NG となり **F3 へ繰越**。

## 概要

可能な限り `npx tauri dev` 上での手動操作で確認する。ユニット／コンポーネントテストは vitest。F1 スコープ外（受け入れ基準 6〜13, 15）はここでは確認しない。

## 前提条件

- `feature/prompt-edit-palette` ブランチをチェックアウト済み
- `npm install` 済み
- `npx tauri dev` でアプリが起動している
- 任意のプロジェクトフォルダを開き、ターミナルタブを最低 1 つ起動してある
- macOS を主対象とする（Win/Linux は送信ショートカットの分岐のみ別途確認）

## 手動テスト

### ケース 1: タブ右クリックからパレット起動（受け入れ基準 1）

**手順:**

1. 任意のターミナルタブを右クリックする。
2. コンテキストメニュー先頭の「プロンプトを編集...」を選択する。

**期待結果:**

- 中央にモーダルが開く
- ヘッダに「プロンプトを編集 → {タブ名} に送信」と表示される
- textarea にフォーカスが入っており、即座に文字入力できる

**確認結果:**

- [x] OK（2026-04-18 確認）

### ケース 8: グローバルショートカットが発火しない（受け入れ基準 14）→ **F3 に繰越**

**状態:** **NG（F1 で未達成、F3-3 で対応予定）**

2026-04-18 の手動 E2E で、`AppLayout.tsx` のキャプチャフェーズ keydown リスナが textarea の `stopPropagation` より先に発火するため、パレット表示中に `Ctrl+Tab` / `F2` 等が抑止できないことを確認した。F1 ではこれ以上対応せず、F3-3（グローバルショートカット競合の整理）で `AppLayout` 側に `isOpen` の早期 return を集約して解決する。

**元の手順（参考、F3 の動作確認用）:**

1. パレットを開く。
2. textarea にフォーカスがある状態で `Ctrl+Tab` を押す（本来コンテンツ/ターミナル切替）。
3. `F2` を押す（本来タブリネーム）。

**期待結果（F3 達成時）:**

- 画面は切り替わらない（パレット表示中はメインタブ切替が発火しない）
- タブリネームインライン編集が開かない
- textarea の入力のみが反応する

**F1 確認結果:** NG（既知・F3 に繰越）

### ケース 9: アクティブタブ無しでは起動しない

**手順:**

1. すべてのターミナルタブを閉じる。
2. `Cmd+Shift+P` を押す。
3. 右クリックメニューに「プロンプトを編集...」が出る状況か確認する（タブが無ければそもそもコンテキストメニューが無い）。

**期待結果:**

- パレットは開かない（no-op）
- コンソールエラーが出ない

**確認結果:**

- [x] OK（2026-04-18 確認）

---

### ケース 10: パレット表示中の既存パス挿入（回帰確認、F1 スコープ）

**手順:**

1. パレットを開いた状態でファイルツリーのノードを `Cmd+Click` する。

**期待結果（F1 時点）:**

- ツリーノードのクリック＝パレット外クリックにより、Radix Dialog 既定の `onPointerDownOutside` でパレットが閉じる。
- パレットが閉じた後の `Cmd+Click` は **既存挙動どおり** アクティブターミナルへ直接書き込まれる（パレットには入らない）。
- F1 ではこの挙動を **許容仕様** とする（パレット外クリックで閉じること自体はモーダル UX として妥当）。F2 でパス挿入ディスパッチを切り替える際に、「パレット開時は閉じず textarea に挿入する」へ整理する。

**確認結果:**

- [x] OK（F1 の許容仕様）
