# テスト手順書 - notification-click-activate

## 概要

OS 通知クリックによる「発信元ウィンドウ＋ターミナルタブのアクティブ化」機能の検証手順。
基本は手動テスト。Rust 側のセレクタ・ヘルパーは `cargo test` で、フロントの `findLocationByPtyId` は Vitest で押さえる。

## 前提条件

- `npx tauri dev` で SDDesk が起動できる
- macOS の通知設定で SDDesk からの通知が許可されている（初回起動時にダイアログ確認）
- Claude Code CLI（`claude`）がインストールされ、ターミナル内から実行可能
- Claude Code が OSC 9 通知を発火する状態（`TERM_PROGRAM=iTerm.app` が `pty.rs` 経由で渡されている）

## 手動テスト

### ケース 1: 単一ウィンドウ・単一タブで通知クリック

**手順:**

1. SDDesk を起動する（メインウィンドウのみ）
2. 任意のプロジェクトを開く
3. 新規ターミナルタブを 1 枚開いて Claude Code を起動 (`claude`)
4. メインモードを「コンテンツ」に切り替えてから、SDDesk ウィンドウを最小化
5. Claude Code に承認を要するコマンド（例: ファイル編集）を実行させ、通知を発火させる
6. OS 通知センターから通知をクリックする

**期待結果:**

- SDDesk ウィンドウが最小化から復帰し、フォアグラウンドになる
- メインモードが「ターミナル」に自動切替される
- Claude Code を起動したターミナルタブがアクティブになる
- 該当タブの未読マーク（琥珀ドット）がクリアされる

**確認結果:**

- [ ] OK / NG （備考: ）

### ケース 2: 複数ウィンドウ環境で発信元ウィンドウだけが復帰する

**手順:**

1. SDDesk を起動し、`⌘N`（File > New Window）で 2 つ目のウィンドウ（プロジェクト B）を開く
2. ウィンドウ A（プロジェクト A）でターミナルタブを開き Claude Code を起動
3. ウィンドウ B にフォーカスを移す（A は背面・別ワークスペースでも可）
4. ウィンドウ A の Claude Code から通知を発火
5. 通知をクリック

**期待結果:**

- ウィンドウ A が前面化され、ウィンドウ B はそのまま背面に残る
- ウィンドウ A 内で発信元タブがアクティブ化されている
- ウィンドウ B には `notification-activate` の影響がなく、選択中タブが変わらない

**確認結果:**

- [ ] OK / NG （備考: ）

### ケース 3: コンテンツモード表示中に通知クリックでターミナルモードへ切替

**手順:**

1. ターミナルタブ（複数）で Claude Code を起動した状態で、メインモードを「コンテンツ」に切替
2. 任意の Markdown ファイルを開いてプレビュー表示にする
3. 別アプリにフォーカスを移し、ターミナル A から通知を発火
4. 通知をクリック

**期待結果:**

- メインモードがターミナルへ切替わる
- ターミナル A のタブが選択されている
- コンテンツモードに開いていた Markdown のスクロール位置やタブ構成が破棄されない（戻れば元の状態）

**確認結果:**

- [ ] OK / NG （備考: ）

### ケース 4: フォーカス中＋発信元タブがアクティブな場合は通知が出ない（既存挙動）

**手順:**

1. 1 ウィンドウ + 1 タブ環境で Claude Code を起動
2. SDDesk をフォアグラウンドのまま、当該タブをアクティブにする
3. Claude Code に通知を発火させる

**期待結果:**

- OS 通知が出ない
- `notification-activate` も発火しない（既に表示中のため）
- アプリが意図せず再フォーカスしない

**確認結果:**

- [ ] OK / NG （備考: ）

---

### ケース 5: 分割表示で発信元タブが secondary ペインにある場合

**手順:**

1. ターミナルパネルの分割を有効化（`splitEnabled = true`）
2. primary ペインに「Terminal 1」、secondary ペインに「Terminal 2」を配置し、Terminal 2 で Claude Code 起動
3. focus を primary ペイン（Terminal 1）に置き、別アプリへフォーカス移動
4. Terminal 2 から通知発火 → 通知クリック

**期待結果:**

- ウィンドウが復帰する
- secondary ペインがアクティブになり、Terminal 2 が選択される
- `setFocusedPane('secondary')` が反映され、xterm 入力が secondary 側に向く

**確認結果:**

- [ ] OK / NG （備考: ）

### ケース 6: PTY 終了済みのタブを指す通知をクリックした場合

**手順:**

1. ターミナルで Claude Code を起動 → 通知を発火させる
2. すぐに `exit` でシェルを終了させる（`handlePtyExited` でタブが入れ替わる）
3. その後 OS 通知をクリック

**期待結果:**

- ウィンドウ復帰は行われる
- `findLocationByPtyId` が null を返し、タブ切替は行われない（クラッシュ・例外も出ない）
- ログ or console に warning が残る程度に留まる

**確認結果:**

- [ ] OK / NG （備考: ）

## 自動テスト

### フロントエンドテスト（Vitest）

```bash
npm run test
```

確認するテスト:

- `terminalStore.findLocationByPtyId` がプライマリ／セカンダリ両ペインのタブを正しく逆引きする
- 該当 PTY が存在しない場合に `null` を返す

### 型チェック

```bash
npm run build
```

### Rustテスト

```bash
cd src-tauri && cargo test
```

確認するテスト:

- `NotificationClickTargets::register` → `take` の往復が成立する
- 同一 ID の `take` を 2 回呼ぶと 2 回目は `None`
- `is_window_focused(label)` が未知ラベルで `false` を返す

### Rustの型チェック

```bash
cd src-tauri && cargo check
```

## 回帰テスト

既存機能への影響がないことを確認する。

- [ ] ファイルツリーが正常に表示・展開できる
- [ ] ターミナルタブの追加 / 削除 / リネームが正常に動作する
- [ ] 既存の OSC 9 未読マーク（琥珀ドット）が引き続き付与される
- [ ] `⌘N` での新規ウィンドウ生成が二重発火しない（`menu-new-window` ガードに副作用がない）
- [ ] HTTP `/claude-hook/{event}` 経由の通知が引き続き発火する（`pty_id` 解決は本要件スコープ外なのでウィンドウ復帰のみ動作すれば OK）
- [ ] Claude Code 通知 ON/OFF トグルで通知抑制が機能する
