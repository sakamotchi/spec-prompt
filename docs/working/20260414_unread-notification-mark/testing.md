# テスト手順書 - unread-notification-mark（P4）

## 概要

P4（未読通知マーク / タブ強調表示）の手動テストおよび自動テスト手順。通知発火 → マーク付与 → ユーザー復帰 → マーク解除のライフサイクルが期待通りに動くことを確認する。

## 前提条件

- `npx tauri dev` でアプリが起動していること
- Phase 1〜3 の機能がすべて動作していること（通知タイトル差し込み / OSC 動的タイトル / 手動リネーム）
- アプリ内設定で「Claude Code 通知」が ON
- macOS のシステム設定で SpecPrompt の通知が許可されている

---

## 手動テスト

### ケース 1: バックグラウンドで通知発火 → マーク付与

**手順:**

1. アプリを起動し、タブを 2 つに増やす（`Terminal 1`, `Terminal 2`）
2. タブ 2 をアクティブにする（`Terminal 1` は非アクティブ）
3. アプリを Cmd+H でバックグラウンドにする
4. 別の手段（iTerm 等）で SpecPrompt の `Terminal 1` に入り、`printf '\033]9;tab1 notification\007'` を送るか、claude を操作して承認要求を発生させる
   - 実運用では直接タブ 1 内で OSC 9 を送るのが簡単。バックグラウンド中にタブ 1 を選んで入力できないため、事前に `sleep 5 && printf '\033]9;tab1\007'` のように仕込んでおくか、直接 `claude` で承認要求を出させる
5. 通知が OS 右上に表示されることを確認
6. アプリをフォアグラウンドに戻す（Cmd+Tab 等）**が、タブ切替はまだしない**

**期待結果:**

- タブ 1 のラベル前に `●` が表示される
- タブ 1 の背景に薄いアクセントハイライト
- タブ 1 の左ボーダーがアクセント色
- タブ 2 は変化なし

**確認結果:**

- [ ] OK / NG

### ケース 2: マーク付きタブをクリックして解除

**手順:**

1. ケース 1 の続き（タブ 1 に `●` 付き、タブ 2 がアクティブ、アプリはフォアグラウンド）
2. タブ 1 をクリックしてアクティブ化

**期待結果:**

- タブ 1 がアクティブになる
- タブ 1 の `●` / ハイライト / 左ボーダーが即座に消える

**確認結果:**

- [ ] OK / NG

---

### ケース 3: アプリフォーカス復帰時、アクティブタブがマーク付きなら解除

**手順:**

1. タブ 1 をアクティブにしてから、別タブで通知発火。これは難しいので以下の代替:
   - タブ 2 をアクティブ → Cmd+H → タブ 2 で通知発火（フォアグラウンド復帰前に、タブ 2 がアクティブのまま）
2. Cmd+H でバックグラウンド
3. 別途の方法（別ターミナル or タイマー）でタブ 2 に OSC 9 を送り、通知を発火させる
4. 通知が届いた直後、アプリをフォアグラウンドに戻す（Cmd+Tab）

**期待結果:**

- 復帰した瞬間、アクティブだったタブ 2 の `●` / ハイライトが解除されている（復帰時に `onFocusChanged` で自動 clear されるため、ユーザーは `●` を見ないか、一瞬だけ見える）

**確認結果:**

- [ ] OK / NG

### ケース 4: アクティブじゃないタブのマークは復帰だけでは消えない

**期待結果:**

- タブ 2 に `●` / ハイライトが残っている
- タブ 1 は変化なし
- タブ 2 をクリックすれば解除される（ケース 2 と同じ挙動）

**確認結果:**

- [ ] OK / NG

### ケース 6: 同じタブで連続通知 → state 不変

**手順:**

1. タブ 1 を非アクティブにして Cmd+H
2. タブ 1 で `printf '\033]9;first\007'`（バックグラウンド中に仕込むのが難しければ、何らかの手段で連発させる）
3. タブ 1 で `printf '\033]9;second\007'`（連続）

**期待結果:**

- 1 回目で `●` が立ち、2 回目でも `●` のまま（既に立っているので変化なし）
- DevTools などでストアを観察し、state 参照が変化しないこと（開発者確認。通常ユーザーは見えない）

**確認結果:**

- [ ] OK / NG（目視できる範囲で OK なら可）

## 自動テスト

### フロントエンドテスト（Vitest）

```bash
npm run test
```

以下が新規に追加されパスすること:

- `markUnread`: 非アクティブタブに true セット / アクティブ+フォーカス時 no-op / 未知 pty_id 無視 / 連続呼び出しで state 参照不変
- `clearUnread`: true → false / もともと false で state 参照不変

```bash
npm run build
```

```bash
cd src-tauri && cargo test
```

既存 99 件全件パス（P4 では emit 追加のみのため新規テストなし）。
